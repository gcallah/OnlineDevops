A Guide to Our DevOps Projects

Our guiding principle:
The goal of DevOps is to be able to rapidly and reliably deliver new features to customers, thus delivering value and enabling rapid feedback and learning.

To that end, we adopt the following practices:

  • Work on all projects should proceed by regular, incremental improvements. The "school model" of waiting until the end of the semester and doing everything at once will result in significant points lost. (It also results in significantly worse outcomes, which is the real point here.)
  • Tasks in a project should appear on a kanban board.
  • Project discussions take place on Slack.
  • Every project should be under version control using git.
  • All relevant computer files relating to the project should be under version control, not just source code. This includes statements of goals, documentation, other sources used in the project, configuration scripts, List of dependencies, diagrams.
  • All local products that can be built should be built with a makefile.
  • Every project should have an extensive, automated test coverage.
  • Every project should be deployed with scripts and configuration files, not things run by hand and stored in participants' heads.
  • Each project should conduct regular code reviews.
  • We should use automated linting.
  • Commits should trigger a test so that testing is done even if an individual programmer tries to skip it. Code should be pushed onto production servers only if these tests pass. We currently use Travis to that end.
  • Each project should be deployable in the Docker container, if possible.
  • If the project involves a server that should run on an ongoing basis, that server should be monitored.
